# Cross-Functional Collaboration Checklist

## Purpose
This checklist helps project teams ensure clear communication and alignment across all personas and roles throughout the project lifecycle.

## Project Initiation

### Kickoff Meeting
- [ ] All core personas identified for this project
- [ ] Additional personas (UX/UI, DevOps, Business Analyst, Support) assessed for inclusion
- [ ] RACI matrix created (Responsible, Accountable, Consulted, Informed)
- [ ] Communication cadence agreed upon
- [ ] Escalation paths documented
- [ ] Success metrics defined and shared with all roles

### Stakeholder Alignment
- [ ] Business requirements validated by Business Analyst and Product Manager
- [ ] Technical feasibility reviewed by Developers and DevOps Engineer
- [ ] User experience goals confirmed with UX/UI Designer
- [ ] Support/Customer Success perspective incorporated
- [ ] Dependencies and risks identified across all teams

---

## Project Planning

### Requirements Definition
- [ ] User stories include acceptance criteria co-authored by Product Manager and QA
- [ ] Technical requirements reviewed by Developers and DevOps Engineer
- [ ] UX/UI considerations documented and shared with team
- [ ] Business Analyst confirms all requirements are clear and complete
- [ ] Support team provides input on known customer pain points

### Design & Technical Planning
- [ ] Design reviews scheduled with Developers and Project Manager
- [ ] Deployment architecture reviewed by DevOps Engineer
- [ ] Test strategy defined collaboratively with QA
- [ ] Integration points with other teams identified
- [ ] Rollback and contingency plans documented

### Timeline & Milestones
- [ ] Project Manager confirms timeline is achievable
- [ ] All personas confirm their capacity and dependencies
- [ ] Release window coordinated with Support team
- [ ] Marketing/Communication timelines aligned if applicable

---

## Execution Phase

### Sprint Planning
- [ ] Backlog items have clear acceptance criteria (Product Manager + QA)
- [ ] Technical tasks include DevOps and infrastructure requirements
- [ ] UX/UI tasks identified and sequenced appropriately
- [ ] Dependencies flagged across personas
- [ ] Team capacity verified across all roles

### Daily Standups
- [ ] Progress updates from each persona represented
- [ ] Blockers and dependencies raised early
- [ ] Cross-functional issues escalated immediately
- [ ] Next day handoffs confirmed (e.g., Dev to QA)

### Code & Design Reviews
- [ ] Developers participate in design reviews before implementation
- [ ] QA reviews acceptance criteria before development begins
- [ ] DevOps reviews code for deployment and monitoring considerations
- [ ] Design reviews include technical feasibility input from Developers
- [ ] Security scanning and best practices confirmed

### Quality Assurance
- [ ] QA confirms all acceptance criteria are testable
- [ ] Test environment set up by DevOps with QA oversight
- [ ] Defects logged with clear reproduction steps and owner assignment
- [ ] Developers and QA collaborate on root cause analysis
- [ ] Regression testing coordinated before release

---

## Release Preparation

### Pre-Release Validation
- [ ] Product Manager confirms feature meets success metrics
- [ ] Developers confirm code quality and test coverage targets met
- [ ] QA confirms all acceptance criteria passed and risks mitigated
- [ ] DevOps confirms deployment runbook is complete and tested
- [ ] Support team trained on new features and prepared for escalations
- [ ] Release notes drafted and reviewed by Product Manager and QA

### Release Readiness
- [ ] Deployment window scheduled and communicated to all teams
- [ ] Rollback plan documented and tested by DevOps
- [ ] On-call support roster confirmed
- [ ] Communication plan finalized (announcement, blog, customer updates)
- [ ] Smoke test checklist prepared with QA

### Deployment
- [ ] DevOps executes deployment with monitoring from all teams
- [ ] QA runs smoke tests and verifies critical paths
- [ ] Support team monitors for customer-reported issues
- [ ] Product Manager tracks usage and initial metrics
- [ ] Project Manager coordinates any escalations

---

## Post-Release & Continuous Improvement

### Release Verification
- [ ] Metrics tracking confirms success criteria are being met
- [ ] Support team reports on customer feedback and issues
- [ ] Developers monitor error rates and performance
- [ ] QA tracks defects reported in production
- [ ] DevOps confirms system stability and resource utilization

### Retrospective
- [ ] All personas attend retrospective meeting
- [ ] What went well from each perspective documented
- [ ] Improvement opportunities identified across teams
- [ ] Action items assigned with clear owners and deadlines
- [ ] Follow-up on previous action items tracked

### Follow-Up Actions
- [ ] Process improvements implemented for next project
- [ ] Documentation updated based on learnings
- [ ] Best practices shared across teams
- [ ] Metrics and outcomes captured in project record

---

## Communication Best Practices Across Personas

### For Project Managers
- Facilitate regular syncs that include all relevant personas
- Create a single source of truth (shared document or project board)
- Flag dependencies proactively and early
- Ensure escalation paths are clear and respected

### For Product Managers
- Share context on "why" so all personas understand the bigger picture
- Involve technical teams early in discovery and prioritization
- Validate acceptance criteria with all stakeholders before work begins
- Measure and share impact post-release with the full team

### For Developers
- Communicate technical constraints early
- Ask clarifying questions rather than assuming requirements
- Participate in design reviews and planning sessions
- Share progress and blockers in standups

### For QA/Testing
- Define testability requirements early with Developers
- Communicate test plans and timeline with Project Manager
- Escalate quality risks proactively
- Partner with Support on post-release issue triage

### For UX/UI Designers
- Communicate design decisions and rationale to Developers
- Participate in feasibility discussions early
- Provide design system guidance and component specs
- Validate implementation matches design intent

### For DevOps Engineers
- Share infrastructure requirements and constraints early
- Participate in deployment planning with Project Manager
- Communicate monitoring and alerting status during releases
- Document operational runbooks collaboratively with Developers

### For Business Analysts
- Clarify requirements early and often with Product Manager
- Ensure all personas have input into requirements
- Document scope clearly and flag out-of-scope items
- Validate acceptance criteria with QA before work begins

### For Support/Customer Success
- Provide customer context and feedback to Product Manager
- Escalate product issues with clear severity and customer impact
- Communicate product changes to customers proactively
- Share customer satisfaction metrics with Product team

---

## Red Flags to Watch For

- ⚠️ Requirements unclear or changing frequently without Business Analyst involvement
- ⚠️ Design decisions made without Developer feasibility input
- ⚠️ Testing strategy not defined before development begins
- ⚠️ DevOps involvement missing from deployment planning
- ⚠️ Support team unaware of upcoming features before release
- ⚠️ Risk register not reviewed with all personas weekly
- ⚠️ Communication happening in silos instead of cross-functional syncs
- ⚠️ Release timeline set without confirming all personas' capacity
